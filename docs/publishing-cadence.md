# 公開頻度ルール

AI Compass Journalは量より再現性を優先します。noteは毎日動かしてよいですが、HPは完成版の辞典として薄い記事を増やしません。

## 基本ルール

- 下書きは毎日1本まで
- 公開は最大1日1本まで
- 通常は週3本までユースケース詳細を公開
- 週1本は既存ページの強化に使う
- 月1回は無料キット、プロンプト、ワークフロー、articleBacklogをまとめて改善
- 80点未満の記事は公開しない

## 毎日

目的: note調査からHP候補を増やす。

作業:
- noteテーマを1件選ぶ
- `src/data/articleBacklog.ts` に候補を1件まで追加
- 既存候補との重複を確認する
- 公開は急がず、候補の質を上げる

公開判断:
- その日に公開してよいのは最大1本
- 80点未満なら候補追加だけで止める

## 週3回

目的: 準備中ユースケースを詳細ページへ昇格する。

作業:
- articleBacklogまたは準備中ユースケースから1件選ぶ
- 手順、プロンプト、入力例、出力例、確認ポイント、失敗例を入れる
- `/updates` に記録する
- lint / typecheck / buildを通す

公開判断:
- 読者の作業が1つに絞れている
- `/free` `/prompts` `/workflows` `/guides` に接続できる
- 品質スコア80点以上

## 週1回

目的: 既存ページを保存版として強化する。

作業:
- 表示回数、内部リンク、読者導線を見て強化対象を選ぶ
- 入力例、出力例、改善プロンプト、よくある失敗を足す
- 古い表現や未接続サービス表現を見直す

公開判断:
- 新規ページを増やすより既存ページ強化の方が価値が高い週は、追加公開を控える

## 月1回

目的: サイト全体の導線と無料資産を改善する。

作業:
- `public/free-starter-kit` の内容を見直す
- `/free` `/prompts` `/workflows` `/guides` の矛盾を消す
- `articleBacklog` の古い候補、重複候補を整理する
- 商品、メルマガ、BOOTHなど未接続サービスの表現を確認する

## 公開しない判断

次のどれかに当てはまる場合は公開しません。

- 品質スコアが80点未満
- note記事の丸ごと転載になっている
- ツール紹介やAIニュース寄りで、手順が弱い
- プロンプトが抽象的で使えない
- 入力例・出力例がない
- 未確認情報を断定している
- 内部リンクが不足している
- `/updates` に記録していない

## PR運用

- 1PRにつき公開ページは原則1本
- 大きな変更は、候補追加、詳細化、改善、README/docs更新に分ける
- GitHub PRは人間が確認してからマージする
- Vercel、BOOTH、メルマガ、Search Consoleなど外部サービス操作はPRに含めない
