# AI Compass Journal ロードマップ

## 最終目標

AI Compass Journal の最終目標は、次の4つを同時に満たすことです。

1. AI活用の実践辞典になる
   - 読者が「何をしたいか」から逆引きできる
   - 記事ごとに再現手順、プロンプト、入力例、出力例、確認ポイントが揃っている

2. 独自実験メディアになる
   - 一般論ではなく、実際に試した自動化・運用実験の記録を蓄積する
   - 成功だけでなく失敗、制約、改善余地まで含めて公開する

3. 半自動 / 自動更新できるAI運用サイトになる
   - Codex、ChatGPT、Claude などを使い分けて、調査、下書き、実装、レビュー、公開準備を回す
   - 人間は本当に境界判断が必要な箇所に集中する

4. note転載サイトではなく、HP専用の資産を置く
   - note は実験ログ
   - HP は再現手順、プロンプト、テンプレート、導線の保存版
   - そのままコピペ・再利用できる実務向けの形にする

## 現在地

現時点の AI Compass Journal は、場当たり的な更新フェーズを抜けつつあり、基盤自体はかなり整っています。

- サイト基盤は完成度が高い
- Vercel 自動デプロイ済み
- `/ai-use-cases` 運用開始済み
- `/experiments` に初回実験 `run-now-automation-test` 公開済み
- `find-routine-automation` など詳細ページ追加済み
- 完全自動更新ではなく、現状は半自動運用

つまり、サイトの骨組みと初期コンテンツ運用は成立している一方で、運用OS、レビュー体制、権限まわり、導線、計測がまだ揃いきっていません。

## 未達成項目

次の項目は未達成であり、今後の優先対応対象です。

- Search Console 未登録
- GitHub push / PR 作成の自動化未達成
- note から HP への導線未整備
- Author Block 未実装
- 継続レビュー体制未整備
- マネタイズ未着手

これらは個別に見ると小さく見えますが、実際には「更新を継続できるか」「資産として積み上がるか」に直結するため、後回しにしすぎない方がよい項目です。

## 優先順位

### Phase 1: 運用OS整備

最優先は、更新を安定して回すための運用設計を整えることです。

- ロードマップの明文化
- エージェントレビュー体制の整理
- Codex へ投げる標準タスク形式の固定
- PR テンプレートの標準化
- 次にやる PR 候補の見える化

このフェーズを先に行うことで、今後の更新が「毎回ゼロから考える作業」ではなくなります。

### Phase 2: Search Console 対応

次に優先すべきは Search Console の導入です。

- verification 方法の整理
- セットアップ手順の文書化
- index 対象と除外対象の整理
- SEO レビュー時の確認観点の整備

計測なしでSEO改善を進めると、改善判断が感覚論に寄りやすいため、早めの対応が必要です。

### Phase 3: Codex / GitHub 権限問題の解消

半自動更新のボトルネックになっている GitHub 認証・権限問題を解消します。

- push 失敗原因の切り分け
- PR 作成失敗原因の切り分け
- fresh clone 運用との相性確認
- 認証経路の標準化

ここが解消されると、ローカル実装から PR 作成までの自動化率を引き上げやすくなります。

### Phase 4: Author Block / note CTA

記事単位の導線強化フェーズです。

- Author Block 実装
- note への導線テンプレート整備
- HP → note / note → HP の役割分担の明確化

このフェーズでは、読者が「読むだけ」で終わらず、他ページや媒体に自然に流れる導線を整えます。

### Phase 5: 独自実験2本目

`run-now-automation-test` に続く2本目の実験記事を公開し、独自実験メディアとしての軸を固めます。

- 仮説
- 実測値
- 失敗ケース
- 自動化率
- 次に試すこと

この形式を再利用できるようにして、実験記事の量産ではなく「型の定着」を優先します。

### Phase 6: 無料キット / 商品 / マネタイズ

最後にマネタイズ導線の整備へ進みます。

- 無料キットの訴求改善
- 商品一覧の整理
- プレースホルダー段階の戦略設計
- 相談・商品・アフィリエイトの線引き

ここは土台が整ってから着手する方が、導線の一貫性を壊しにくく、運用も楽になります。

## 運用原則

- note の転載をしない
- HP 専用の再現手順と導線を作る
- 未確認情報を断定しない
- エージェントに任せる部分と人間が判断する部分を分ける
- 自動化率を上げても、公開境界の判断は完全自動化しない
